HTTPS为什么安全

HTTPS简介

HTTPS 不是一个新鲜的协议，它仍然是基于 HTTP 协议，可以把 HTTPS 协议看成是穿着防弹衣的 HTTP 协议，协议层对比如下图：

HTTP 由于是明文传输，所以安全上存在以下三个风险：

  • 窃听风险，比如通信链路上可以获取通信内容，用户号容易没。
  • 篡改风险，比如强制入垃圾广告，视觉污染，用户眼容易瞎。
  • 冒充风险，比如冒充淘宝网站，用户钱容易没

基于 HTTP 协议的网络传输是明文的，对响应的内容也不加验证，这意味着只要传输内容被黑客截获，黑客就能知道传输内容，甚至能够返回一段修改后的内容给用户。这就是常说的 HTTP 劫持，一些运营商会使用这种手段往用户的网页里注入广告。

加密

对称加密

对称加密，顾名思义，加密方与解密方使用同一钥匙(秘钥)。具体一些就是，发送方通过使用相应的加密算法和秘钥，对将要发送的信息进行加密；对于接收方而言，使用解密算法和相同的秘钥解锁信息，从而有能力阅读信息。

  • 优点：很快，方便
  • 缺点： A给B发送密钥时容易被截取

非对称加密

非对称加密使用两个密钥来进行加密和解密，公开密钥是所有人都知道的密钥，私有密钥是持有方才知道的密钥，一般来说私钥就放在服务器里，数据经过公钥加密就只能被私钥解密，数据经过私钥加密就只能被公钥解密。

如果应用到客户端和服务端的话，就是服务端自己拥有成对的私钥和公钥，然后公布自己的公钥让客户知道，客户端用公钥把自己的数据进行加密，加密后用公钥反而无法解密这段数据，一定要用服务端的私钥才能解密，这样的非对称加密其实也被叫做公钥加密。因为不同于对称加密中只使用一把私钥进行加密。当然是通过算法来实现的。

证书

HTTPS 协议需要向 CA（证书权威机构）申请数字证书，来保证服务器的身份是可信的。

这里还有一个问题，虽然我们现在可以对数据进行加密，但是我们还是不知道和我们沟通的是否是自己想要沟通的对象，比如b站的域名

就会有不法分子伪装b站域名来让你访问，还好ＨＴＴＰＳ解决了这个问题。因为服务端需要申请ＳＳＬ证书来证明这个域名是大家所熟悉的Ｂ站（ＳＳＬ证书比ＴＳＬ证书更为人所熟知）。

ＳＳＬ证书其实就是保存在源服务器的数据文件，要让ＳＳＬ证书生效就需要想ＣＡ申请，也就是Certificate Authority证书授权中心，ＣＡ属于第三方机构，这个机构颁发的证书除了表明域名是属于谁的，日期等等信息以外，重要的是这个证书里还包括了特定的公钥和私钥。

简单来说服务器安装了ＳＳＬ证书以后，用户就可以通过ＨＴＴＰＳ来访问服务器了，当然浏览器也会把ＨＴＴＰ默认的端口80改为ＨＴＴＰＳ的默认端口４４３。

TLS握手

那现在浏览器通过ＨＴＴＰＳ访问服务器具体会有哪些变化呢？

这里以ＴＬＳ１.２版本为例。

过程

正常的ＴＣＰ三次握手是不变的

使用Wireshark进行抓包分析，抓包界面如下

ＴＣＰ三次握手之后，客户端发送了一个Client　Ｈｅｌｌｏ给服务端

客户端就会告诉服务端支持ＴＬＳ１.２版本和支持的加密套件，这里的16个加密套件组合可以理解为不同的加密算法组合，然后生成一个随机数发送给服务端（第二个框框就是随机数）。

服务端收到客户端打的招呼，当然也要和客户端打招呼，所以接下来就是Ｓｅｒｖｅｒ　Ｈｅｌｌｏ发送给客户端。在响应报文里面会告诉客户端，服务端确认支持的ＴＬＳ版本以及选择的加密套件，并且服务器也生成一个随机数发送给客户端。

接下服务器会再发一个响应，来出示服务器自己的证书，这样浏览器就可以根据对照自己的证书信任列表来确认这个服务器是否可信。

发了证书，当然少不了哦公钥。Ｓｅｒｖｅｒ　ｋｅｙ这一步就会把公钥发送给客户端，而不会傻傻的吧私钥发出去。如果服务器也想要客户端的证书会在这里发出请求，比如登录网银就很可能需要这个步骤了。

服务器发生了这么多响应，还要告诉客户端发送完了。

截止到目前这些请求和响应依旧还未进行加密，现在就轮到客户端来处理这些响应了。

Ｃｌｉｅｎｔ　ｋｅｙ　Ｅｘｃｈａｎｇｅ这一步是重点，这一步客户端会生成第三个随机数，也叫预主密钥，这第三个随机数会用刚刚收到的公钥进行加密，并且把这个加密后的随机数发送给服务器，正是这里Pubkev显示的随机数。

ｃｈａｎｇｅ　ｃｉｐｈｅｒ　Ｓｐｅｃ这一步就是告诉服务器往后的数据就用商议好的算法和密钥来加密咯

客户端这里发送Ｅｎｃｒｙｐｔｅｄ　Ｈａｎｄｓｈａｋｅ　Ｍｅｓｓａｇｅ，表示客户端这边的TLS协商已经没有问题了，加密开始。

同样地后面服务器也发来了Ｅｎｃｒｙｐｔｅｄ　Ｈａｎｄｓｈａｋｅ　Ｍｅｓｓａｇｅ，表示服务器这边也准备好了，这里也表示着TLS的握手已经成功了，可以给数据加密讲行交换了。

这里没有解释第一随机数和第二随机数的含义，总结时会讲到

握手总结

  • 客户端和服务端打招呼，并且把自己支持的TLS版本，加密套件发给服务端，同时还生成了一个随机数给服务端，称为第一随机数
  • 接着服务端打招呼，服务端确认支持的TLS版本以及选择的加密套件，并且服务器也生成随机数发给客户端，称为第二随机数
  • 接着服务端还把证书和公钥发送给客户端
  • 都发送完毕就告诉客户端
  • 现在客户端这边生成随机数，我们称为第3个随机数，预主密钥，这个预主密钥不会直接发送出去，而是用刚刚收到的公钥进行加密后再发送出去，然后客户端这边的TLS协商已经没问题了，加密开始
  • 服务端收到加密后的预主密钥以后，会用自己的私钥进行解密，这样服务端就知道预主密钥了，而且只有客户端和服务端知道这预主密钥，因为没有进行直接传输，没有其它人知道这预主密钥是什么，除非私钥被泄露了。
  • 最后客户端用预主密钥，第1随机数和第2随机数计算出会话密钥，服务端也用预主密钥，第1随机数和第2随机数计算出会话密钥，各自得到的会话密钥是相同的，也就是说前面的步骤都是非对称加密，为了得到这个会话密钥

后面的会话大家都只使用这个会话密钥对数据进行加密，也就是后面使用的是对称加密，大家都使用同一个密钥，会话后面不使用非对称加密是因为大家都能看到消耗资源非常大，而且得到会话密钥以后就没人知道密钥是什么了，如果与其它服务器沟通就是建立新的会话，会话密钥只应用在当前会话，更加提高了安全性。
