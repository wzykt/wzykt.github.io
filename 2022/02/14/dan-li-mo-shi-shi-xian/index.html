设计模式
学习笔记
发布日期:   2022-02-14
文章字数:   1.9k

单例模式有很多实现方法，饿汉、懒汉、静态内部类、枚举类，试分析每种实现下获取单例对象（即调用 getInstance）时的线程安全，并思考注释中的问题

饿汉式：类加载就会导致该单实例对象被创建

懒汉式：类加载不会导致该单实例对象被创建，而是首次使用该对象时才会创建

实现1懒汉式

// 问题1：为什么加 final
// 问题2：如果实现了序列化接口, 还要做什么来防止反序列化破坏单例
public final class Singleton implements Serializable {
    // 问题3：为什么设置为私有? 是否能防止反射创建新的实例?
    private Singleton() {
    }

    // 问题4：这样初始化是否能保证单例对象创建时的线程安全?
    private static final Singleton INSTANCE = new Singleton();

    // 问题5：为什么提供静态方法而不是直接将 INSTANCE 设置为 public, 说出你知道的理由
    public static Singleton getInstance() {
        return INSTANCE;
    }

    public Object readResolve() {
        return INSTANCE;
    }
}

Q1:避免子类重写父类的方法，导致破坏了单例

Q2：创建对象不只有new的方式，当我们实现了序列化的接口，将来进行反序列化的时候，会生成一个新的对象，反序列化生成的对象和我们所维护的单例对象很明显不是同一个，这也就破坏的单例。

怎么解决呢？重写readResolve()方法，当用反序列化创建对象时，会调用readResovle()，我们直接给它重写返回我们所维护的单例对象，而不是通过反序列化生成的对象。

Q3：用来隐藏类的构造方法。

不能防止反射创建新的实例，反射可以直接拿到的我们的构造器对象，暴力的反射获取新的实例。

Q4：依靠类加载时的clinit静态初始化时执行，JVM保证了线程安全，但对内存不友好。

Q5：用方法相当于提供了更好的封装性。方法可以更好的支持泛型。

实现2枚举

// 问题1：枚举单例是如何限制实例个数的
// 问题2：枚举单例在创建时是否有并发问题
// 问题3：枚举单例能否被反射破坏单例
// 问题4：枚举单例能否被反序列化破坏单例
// 问题5：枚举单例属于懒汉式还是饿汉式
// 问题6：枚举单例如果希望加入一些单例创建时的初始化逻辑该如何做
enum Singleton {
    INSTANCE;
}

Q1：枚举类里面定义的枚举对象，我们定义几个，就相当于有几个对象，相当于枚举类中静态成员变量，我们可以通过反编译的方式查看到字节码文件。

Q2：没有并发问题，枚举类中的对象是静态成员变量，静态变量的初始化实在类加载时进行的。

Q3：不能，枚举类型不能通过newInstance反射获取反射对象。因为Class#newInstance()方法会判断是否是枚举类，如果是会抛出异常

Q4：不能，枚举类默认继承自Enum，Enum实现了Serializable接口，所以枚举类型是可以进行序列化和反序列化的，而Enum在设计之初就考虑到了反序列化破坏单例的问题。

Q5：属于饿汉式

Q6：枚举类中是可以写构造方法的，在构造方法中进行初始化即可。

实现3

public final class Singleton {
    private Singleton() { }
    private static Singleton INSTANCE = null;
    // 分析这里的线程安全, 并说明有什么缺点
    public static synchronized Singleton getInstance() {
        if( INSTANCE != null ){
            return INSTANCE;
        }
        INSTANCE = new Singleton();
        return INSTANCE;
    }
}

Q：synchronized这把锁加在了类对象上Singleton.Class，类对象和我们的静态成员变量是一一对应的，加在类对象上，也就相当于对INSTANCE进行了保护。

synchronized不能加在INSTANCE上，因为INSTANCE可能为null，也就意味着可能没有对应的Monitor对象。

缺点是锁的范围大，每次获取实例时都要加锁，性能低。

实现4DCL

double-checked locking 单例模式

实现特点是：

  • 懒惰实例化
  • 首次使用 getInstance() 才使用 synchronized 加锁，后续使用时无需加锁
  • 有隐含的，但很关键的一点：第一个 if 使用了 INSTANCE 变量，是在同步块之外
public final class Singleton {
    private Singleton() { }
    
    private static volatile Singleton INSTANCE = null;

    
    public static Singleton getInstance() {
        if (INSTANCE != null) {
            return INSTANCE;
        }
        synchronized (Singleton.class) {
            
            if (INSTANCE != null) { 
                return INSTANCE;
            }
            INSTANCE = new Singleton();
            return INSTANCE;
        }
    }
}
public final class Singleton {
    private Singleton() { }
    // 问题1：为什么要加volatile？
    private static volatile？ Singleton INSTANCE = null;
    // 问题2：对比实现3, 说出这样做的意义
    public static Singleton getInstance() {
        if (INSTANCE == null) {
            synchronized (Singleton.class) { 
                // 问题3：为什么还要在这里加为空判断, 之前不是判断过了吗
                if (INSTANCE == null) { // t1
                    INSTANCE = new Singleton();
                }
            }
        }
        return INSTANCE;
    }
}

Q1：volatile可以避免指令的重排序

在多线程环境下，上面的代码是有问题的，getInstance 方法对应的字节码为：

      10: monitorenter

      20: dup
      21: invokespecial #4 // Method "<init>":()V

其中

  • 17 表示创建对象，将对象引用入栈 // new Singleton
  • 20 表示复制一份对象引用 // 引用地址
  • 21 表示利用一个对象引用，调用构造方法
  • 24 表示利用一个对象引用，赋值给 static INSTANCE

也许 jvm 会优化为：先执行 24，再执行 21。如果两个线程 t1，t2 按如下时间序列执行：

关键在于 0: getstatic 这行代码在 monitor 控制之外，它就像之前举例中不守规则的人，可以越过 monitor 读取 INSTANCE 变量的值 。

这时 t1 还未完全将构造方法执行完毕，如果在构造方法中要执行很多初始化操作，那么 t2 拿到的是将是一个未初 始化完毕的单例 。

对 INSTANCE 使用 volatile 修饰即可，可以禁用指令重排，但要注意在 JDK 5 以上的版本的 volatile 才会真正有效。

Q2：性能比实现3要高，加锁的时机不同。

Q3：避免首次创建对象时的并发问题，当两个线程都通过的第一个判断，那么后来的线程会在同步代码块哪里阻塞住，当先来的线程创建完对象后，退出同步代码块，如果没有判空操作，那么后来的线程会再一次创建一个新的对象，破坏了单例。

实现5静态内部类

public final class Singleton {
    private Singleton() { }
    // 问题1：属于懒汉式还是饿汉式
    private static class LazyHolder {
        static final Singleton INSTANCE = new Singleton();
    }
    // 问题2：在创建时是否有并发问题
    public static Singleton getInstance() {
        return LazyHolder.INSTANCE;
    }
}

Q1：懒汉式，因为类加载时懒加载的形式，静态内部类在使用的时候完成类加载，不会因为外部类的加载而加载。这是它和静态代码块和静态变量的区别。

Q2：没有，在类加载时，会由JVM来保证线程安全性的。
